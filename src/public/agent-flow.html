PIPELINE LAYER — CROSS-CUTTING SERVICES (new in Master Improvement)
CORE SERVICE
⚙️ PipelineRunner
src/services/pipeline-runner.ts

Tracks each lead through sequential steps:
INTAKE → ENRICH → BRIEF → PROPOSAL → OUTREACH

States: 🟢 WAITING   🟡 RUNNING   🔴 BLOCKED   ✅ DONE

Wired into:
• Agent-01 / 18 → start() on lead creation
• Agent-02 → advance() after enrichment
• Agent-03 → advance() or block() after brief
• W5 reads getBlocked() for morning briefing

Persists to KB on every state change.
loadFromKB() restores pipelines on restart.
UTILITY
✅ DataValidator
src/utils/data-validator.ts

Checks required fields before each pipeline step:

validateForEnrichment(row)
→ requires: company name

validateForBrief(row)
→ requires: company, show, size, budget

validateForOutreach(row)
→ requires: company, email

Returns { valid, missing[], warnings[] }
Agent-03 calls block() if missing.length > 0

PERSISTENCE
🗄️ KB Persistence Layer
3 systems now write state to Knowledge Base
so they survive Railway restarts:

PipelineRunner → pipeline-state-{leadId}
Saves on every state change. Loads on startup.

ThreadContext → thread-context-{userId}
Debounced 30s per user. Restores sessions <4h old.

Approvals → pending-approval-{id}
Saves metadata on register. Marks EXECUTED after handling.
UPGRADED
🧠 Brain — Pipeline Aware
Brain's system prompt now includes
PIPELINE AWARENESS section:

• Knows WAITING / RUNNING / BLOCKED / DONE states
• Knows when to suggest /resume [company]
• Knows /status shows the full pipeline
• Knows /briefing shows blocked + stale items

When Mo asks "what's happening with X?"

📊 Pipeline Step Flow (per lead)
INTAKE  → Agent-01 (/newlead) or Agent-18 (/checkemails)
  ↓ pipelineRunner.start(leadId, company)
ENRICH → Agent-02 (/enrich company)
  ↓ pipelineRunner.advance(leadId, { dmTitle, readiness })
BRIEF  → Agent-03 (/brief company)
  ↓ pipelineRunner.advance() on success
  ✋ pipelineRunner.block(reason) if fields missing
PROPOSAL → manual (move Trello card)
OUTREACH → Agent-13 or Agent-17

BLOCKED → /status shows it → /resume X re-runs it
🧵 ThreadContext — Persisted
src/services/thread-context.ts

Before: in-memory only — lost on every restart
Now: debounced save to KB every 30s per user

On startup: loadThreadsFromKB()
→ Restores sessions less than 4 hours old
→ Brain knows what Mo was working on before restart

Also tracks activeFocus (current lead/show)
so agents have context across commands
🔐 Approvals — Persisted
src/services/approvals.ts

Before: approval callbacks lost on restart
Mo got "approval not found" with no context

Now:
• registerApproval() → saves metadata to KB
• handleApproval() → marks EXECUTED in KB
• On startup: scans KB for pre-restart pending items
• Mo gets a list of what was pending + re-run instructions

Note: JS callbacks cannot be serialized.
Re-running the command gives a fresh approval token.
🚀 Startup Restore
Runs in src/index.ts after initWorkflowEngine()

On every Railway boot:

1. pipelineRunner.loadFromKB()
→ Restores all non-DONE pipelines from KB

2. loadThreadsFromKB()
→ Restores recent user sessions (<4h old)

3. scanPendingApprovalsFromKB()
→ Finds pre-restart pending approvals
→ Sends Mo a Telegram list with re-run instructions

Result: seamless experience after restarts
